# Silver Memory Product Design

## 핵심 가치

Silver Memory는 고인을 단순히 추모 대상이 아니라, 한 사람의 생애와 관계를 오래 보관하는 디지털 기념관으로 다룹니다.

## MVP 사용자

- 유족: 생애 페이지를 만들고 방명록을 관리합니다.
- 방문자: QR 또는 링크로 들어와 고인의 삶을 보고 추억을 남깁니다.
- 운영자: 유족 계정, 신고, 공개 페이지 품질을 관리합니다.

## 핵심 기능

1. 생애 페이지
   - 이름, 생몰연도, 대표 사진, 한 줄 소개
   - 생애 타임라인
   - 가족이 고른 기억 카드
   - 삶의 키워드

2. 방명록
   - 방문자 이름, 관계, 추억 글
   - 승인 대기 기본값
   - 유족의 승인, 숨김, 고정

3. 유족 편집기
   - 새 추모관 생성
   - 최초 유족 코드 발급과 즉시 편집 화면 진입
   - 기본 정보 수정
   - 대표 사진 업로드
   - 타임라인 항목 추가
   - 기억 카드 추가: 짧은 이야기, 좋아했던 것, 영상/사진 링크
   - 공개 범위 설정
   - 유족 코드 입력
   - 가족 초대 링크 발급, 상태 확인, 회수
   - 처음 설정 체크리스트: 유족 코드 보관, 대표 사진, 타임라인, 기억 카드, QR, 가족 초대
   - 가족 백업: 생애 기록, 방명록, 편집 기록을 JSON 파일로 다운로드
   - 편집 기록 확인
   - 실제 QR SVG 미리보기와 카드 다운로드
   - 디자인 설정: 페이지 분위기, 기억 대상, 페이지 구성 선택과 저장
   - 빠른 기록 작성: 타임라인과 기억 카드를 한 화면에서 묶어 추가
   - 기억 대상 선택: 사람 또는 반려동물
   - 페이지 구성 선택: 기록관형, 앨범형, 편지형, 사진첩형, 스토리형

## 유족 편집 권한

현재 MVP는 memorial별 유족 코드를 사용한다.

- 방문자 생애 페이지와 방명록 작성은 공개 접근을 허용한다.
- 유족 편집, 방명록 승인/숨김/고정, 편집 기록 조회는 유족 코드가 있어야 한다.
- 샘플 memorial의 로컬 코드는 `demo-family-token`이다.
- MVP에서는 가족 초대 링크도 편집 권한 토큰으로 인정한다.
- 가족 초대 링크는 만료일과 회수 상태를 가져야 하며, 유족이 언제든 회수할 수 있어야 한다.
- 초대 토큰으로 편집기를 열면 초대받은 이름을 현재 편집자 라벨로 보여주고, 편집자 이름이 기본값이면 해당 이름으로 편집 기록을 남긴다.
- 운영 버전에서는 Google 계정 로그인, 가족 초대 링크, 권한별 역할로 전환한다.

## 공개 범위 원칙

- 기본값은 링크 공개입니다.
- 전체 공개 피드는 유족이 명시적으로 켰을 때만 허용합니다.
- 방명록은 기본 승인제로 둡니다.
- `가족만` 추모관은 공개 생애 화면을 보여주지 않고, 유족 코드 또는 초대 토큰 입력 화면을 보여줍니다.
- `가족만` 추모관에는 공개 방명록 작성도 허용하지 않습니다.

## 공개 쓰기 보호 원칙

- 새 추모관 생성과 방명록 작성은 로그인 없이 열어두되, IP 기준 기본 rate limit을 둔다.
- 새 추모관 생성 기본값은 1시간 5개, 방명록 작성 기본값은 1시간 20개다.
- rate limit은 초기 운영 보호 장치이며, Google 로그인/신고/차단/Turnstile 도입 전까지 최소한의 남용 방지 역할을 한다.

## 공유 링크 원칙

- QR에는 `#/m/{slug}` 형식의 해시 라우트 링크를 담는다.
- 정적 로컬 서버와 운영 nginx SPA fallback 모두에서 같은 링크 구조를 사용한다.
- 프론트엔드는 현재 slug로 `/api/memory/memorials/{slug}`를 조회한다.

## 데이터 백업 원칙

- 유족은 편집기에서 추모관 데이터를 JSON 파일로 내려받을 수 있어야 한다.
- 백업에는 생애 정보, 타임라인, 기억 카드, 방명록, 초대 목록 요약, 편집 기록을 포함한다.
- 유족 코드와 초대 토큰 원문은 백업 파일에 포함하지 않는다.
- 운영 버전에서는 백업 다운로드와 복구를 로그인한 가족 권한으로 제한한다.

## 디자인 테마 원칙

- 서비스 기본 감성은 공공서비스보다 가족 기록관에 가깝게 둔다.
- 색상, 표면감, 강조색은 CSS 변수 토큰으로 관리한다.
- 유족은 memorial별로 `warm`, `ocean`, `garden`, `graphite`, `blossom`, `paper`, `lake` 중 하나를 선택한다.
- 선택한 테마는 backend `designTheme`으로 저장해 QR 방문자에게도 같은 분위기를 보여준다.

## 템플릿 원칙

- `memorialKind=person`은 생애, 삶의 장면, 남겨진 마음을 중심으로 문구를 구성한다.
- `memorialKind=pet`은 함께한 시간, 함께한 장면, 같이 사랑한 마음을 중심으로 문구를 구성한다.
- `pageTemplate=classic`은 타임라인과 기억 카드를 균형 있게 보여준다.
- `pageTemplate=album`은 기억 카드와 사진형 경험을 먼저 보여준다.
- `pageTemplate=letter`는 방명록과 남겨진 글을 먼저 보여준다.
- `pageTemplate=gallery`는 기억 카드를 여러 장 펼쳐 가족 앨범처럼 보여준다.
- `pageTemplate=story`는 기억 카드 한 장면을 크게 보여주고 넘겨보는 흐름을 강조한다.

## AI 태그 원칙

- AI는 고인의 삶을 대신 판단하지 않고, 유족이 편집하기 쉬운 태그 후보만 제안한다.
- 공개 페이지에 노출되는 태그는 유족이 직접 승인한 항목만 사용한다.
- MVP는 로컬 규칙 기반 추천으로 UX를 먼저 검증한다.
- 운영 버전에서는 생애 글, 기억 카드, 승인된 방명록을 LLM에 전달해 태그 후보와 짧은 요약 초안을 만든다.
- 반려동물 memorial도 같은 구조로 두되, 질병/사망 원인처럼 민감한 표현은 기본 추천에서 제외한다.

## 로그인 원칙

- MVP는 유족 코드로 편집 권한을 보호한다.
- 운영 베타부터는 Google 로그인과 가족 초대 링크를 도입한다.
- 공개 페이지 조회와 방명록 작성은 로그인 없이 가능하게 유지한다.
- 편집, 승인, 숨김, 고정, 편집 기록, QR 다운로드는 로그인한 가족 권한으로 제한한다.
- Google 계정은 사람 식별 수단이고, 실제 권한은 memorial별 가족 권한 테이블에서 판단한다.

## 다음 개발 단계

- 가족 관리자 로그인과 초대 링크
- 이미지 업로드 저장소
- QR 스캐너 실기기 판독 테스트
- 공개 페이지 slug
- 신고/차단/스팸 방지

## 후원/꽃 놓기 구상

후원 기능은 추모의 진정성을 해치지 않도록 "과시형 결제"가 아니라 "작은 마음을 남기는 행동"으로 설계한다.

초기 상품안:

- 작은 꽃: 4달러 또는 4,000원대
- 추모 꽃다발: 20달러 또는 20,000원대
- 유족에게 직접 송금되는 후원과 서비스 운영 후원은 분리 표시

원칙:

- 금액 순위를 노출하지 않는다.
- 후원자는 이름 공개, 익명 공개, 비공개를 선택한다.
- 유족이 후원 기능을 켠 memorial에서만 표시한다.
- 결제 전에는 반드시 "유족에게 전달되는 금액/수수료/서비스 운영비"를 명확히 표시한다.
- 고인의 페이지 본문보다 후원 UI가 더 강하게 보이지 않게 한다.

추천 MVP:

1. 실제 결제 전, "꽃 놓기" UI를 비활성 상태로 노출해 반응을 본다.
2. Stripe/토스페이먼츠 중 하나를 붙여 결제 intent를 만든다.
3. `memory_tributes` 테이블에 꽃 종류, 금액, 메시지, 공개 여부, 결제 상태를 저장한다.
4. 방명록과 별개로 "남겨진 꽃" 영역에 조용히 표시한다.
