# Silver Memory Product Design

## 핵심 가치

Silver Memory는 고인을 단순히 추모 대상이 아니라, 한 사람의 생애와 관계를 오래 보관하는 디지털 기념관으로 다룹니다.

## MVP 사용자

- 유족: 생애 페이지를 만들고 방명록을 관리합니다.
- 방문자: QR 또는 링크로 들어와 고인의 삶을 보고 추억을 남깁니다.
- 운영자: 유족 계정, 신고, 공개 페이지 품질을 관리합니다.

## 핵심 기능

1. 생애 페이지
   - 이름, 생몰연도, 대표 사진, 한 줄 소개
   - 생애 타임라인
   - 가족이 고른 기억 카드
   - 삶의 키워드

2. 방명록
   - 방문자 이름, 관계, 추억 글
   - 승인 대기 기본값
   - 유족의 승인, 숨김, 고정

3. 공지사항/알림판
   - 유족이 방문자에게 전할 공식 안내
   - 추모식 일정, 기록 업데이트, 방명록 작성 요청
   - 고정 공지와 유형 구분: 공지, 일정, 업데이트

4. 유족 편집기
   - 새 추모관 생성
   - 최초 유족 코드 발급과 즉시 편집 화면 진입
   - 기본 정보 수정
   - 대표 사진 업로드
   - 타임라인 항목 추가
   - 기억 카드 추가: 짧은 이야기, 좋아했던 것, 영상/사진 링크
   - 공개 범위 설정
   - 유족 코드 입력
   - 가족 초대 링크 발급, 상태 확인, 회수
   - Google 계정 기반 가족 권한 등록, 역할 변경, 회수
   - 공지사항 추가, 수정, 삭제
   - 처음 설정 체크리스트: 유족 코드 보관, 대표 사진, 타임라인, 기억 카드, QR, 가족 초대
   - 가족 백업: 생애 기록, 공지사항, 방명록, 편집 기록을 JSON 파일로 다운로드
   - 편집 기록 확인
   - 버전 상세 확인: 변경 전/후 스냅샷 조회
   - 실제 QR SVG 미리보기와 카드 다운로드
   - 디자인 설정: 페이지 분위기, 기억 대상, 페이지 구성 선택과 저장
   - 빠른 기록 작성: 타임라인과 기억 카드를 한 화면에서 묶어 추가
   - 기억 대상 선택: 사람 또는 반려동물
   - 페이지 구성 선택: 기록관형, 앨범형, 편지형, 사진첩형, 스토리형

## 유족 편집 권한

현재 MVP는 memorial별 유족 코드와 Google 계정 기반 가족 권한을 함께 사용한다.

- 방문자 생애 페이지와 방명록 작성은 공개 접근을 허용한다.
- 유족 편집, 방명록 승인/숨김/고정, 편집 기록 조회는 유족 코드, 초대 토큰, 등록된 가족 권한 중 하나가 있어야 한다.
- 샘플 memorial의 로컬 코드는 `demo-family-token`이다.
- MVP에서는 가족 초대 링크도 편집 권한 토큰으로 인정한다.
- 가족 초대 링크는 만료일과 회수 상태를 가져야 하며, 유족이 언제든 회수할 수 있어야 한다.
- 초대 토큰으로 편집기를 열면 초대받은 이름을 현재 편집자 라벨로 보여주고, 편집자 이름이 기본값이면 해당 이름으로 편집 기록을 남긴다.
- 유족 코드 또는 초대 토큰을 가진 사용자는 Google 계정을 memorial 가족 권한에 등록할 수 있다.
- 가족 권한 역할은 `owner`, `editor`, `viewer`로 둔다.
- `owner`와 `editor`는 유족 편집기를 열 수 있고, `viewer`는 가족만 공개 추모관 열람만 허용한다.
- 가족 권한 관리는 유족 코드/초대 토큰 또는 `owner` 가족 계정만 수행한다.
- 운영 버전에서는 현재 데모 Google 버튼을 실제 OAuth/Spring Security 인증으로 전환한다.

## 공개 범위 원칙

- 기본값은 링크 공개입니다.
- 전체 공개 피드는 유족이 명시적으로 켰을 때만 허용합니다.
- 공지사항은 유족이 작성한 공식 안내로 보고, 공개 페이지 조회 권한과 같은 범위에서 노출합니다.
- 방명록은 기본 승인제로 둡니다.
- 작성 직후 응답으로 받은 `ownerToken`이 있는 브라우저에서는 방문자가 본인 방명록을 수정/삭제할 수 있습니다.
- Google 로그인 사용자는 `author_user_id`와 현재 사용자 ID를 비교해 본인 방명록을 수정/삭제할 수 있습니다.
- 방문자가 글을 수정하면 다시 승인 대기로 돌려 유족이 변경 내용을 확인한 뒤 공개합니다.
- `가족만` 추모관은 공개 생애 화면을 보여주지 않고, 유족 코드 또는 초대 토큰 입력 화면을 보여줍니다.
- `가족만` 추모관에는 공개 방명록 작성도 허용하지 않습니다.

## 공개 쓰기 보호 원칙

- 새 추모관 생성과 방명록 작성은 로그인 없이 열어두되, IP 기준 기본 rate limit을 둔다.
- 새 추모관 생성 기본값은 1시간 5개, 방명록 작성 기본값은 1시간 20개다.
- rate limit은 초기 운영 보호 장치이며, Google 로그인/신고/차단/Turnstile 도입 전까지 최소한의 남용 방지 역할을 한다.

## 공유 링크 원칙

- QR에는 `#/m/{slug}` 형식의 해시 라우트 링크를 담는다.
- 정적 로컬 서버와 운영 nginx SPA fallback 모두에서 같은 링크 구조를 사용한다.
- 프론트엔드는 현재 slug로 `/api/memory/memorials/{slug}`를 조회한다.

## 데이터 백업 원칙

- 유족은 편집기에서 추모관 데이터를 JSON 파일로 내려받을 수 있어야 한다.
- 백업에는 생애 정보, 타임라인, 기억 카드, 공지사항, 방명록, 초대 목록 요약, 가족 권한 요약, 편집 기록, 버전 상세를 포함한다.
- 유족 코드와 초대 토큰 원문은 백업 파일에 포함하지 않는다.
- 백업 가져오기는 먼저 형식과 개수를 미리 보여주고, MVP에서는 운영 DB를 덮어쓰지 않고 브라우저 임시 복구로만 적용한다.
- 운영 버전에서는 백업 다운로드와 복구를 로그인한 가족 권한으로 제한한다.

## 나무위키형 협업 원칙

- 현재 저장소는 PostgreSQL을 유지한다.
- 생애 기록은 개인정보, 비공개 전환, 삭제 요청, 가족 권한이 중요하므로 Git 저장소형 영구 파일 이력보다 DB 기반 revision이 적합하다.
- `memory_edit_events`는 사람이 읽는 편집 요약 로그로 둔다.
- `memory_content_revisions`는 변경 전/후 JSON 스냅샷을 저장하는 버전 원장으로 둔다.
- 스냅샷에는 콘텐츠 필드만 저장하고 유족 코드, 초대 토큰, 토큰 해시 같은 민감 권한 정보는 저장하지 않는다.
- 다음 단계는 필드별 diff 하이라이트, 특정 버전 복원, 동시 편집 충돌 감지, owner 승인형 되돌리기다.

## 대표 사진 원칙

- 대표 사진은 저장 전 미리보기를 제공해 유족이 방문자 첫 화면을 예상할 수 있게 한다.
- 업로드 가능한 파일은 이미지로 제한하고, 현재 backend 제한에 맞춰 5MB 이하로 안내한다.
- 선택한 사진은 `저장하기`를 누르기 전까지 방문자 화면에 반영되지 않는다는 점을 명확히 표시한다.

## 디자인 테마 원칙

- 서비스 기본 감성은 공공서비스보다 가족 기록관에 가깝게 둔다.
- 색상, 표면감, 강조색은 CSS 변수 토큰으로 관리한다.
- 유족은 memorial별로 `warm`, `ocean`, `garden`, `graphite`, `blossom`, `paper`, `lake` 중 하나를 선택한다.
- 선택한 테마는 backend `designTheme`으로 저장해 QR 방문자에게도 같은 분위기를 보여준다.

## 템플릿 원칙

- `memorialKind=person`은 생애, 삶의 장면, 남겨진 마음을 중심으로 문구를 구성한다.
- `memorialKind=pet`은 함께한 시간, 함께한 장면, 같이 사랑한 마음을 중심으로 문구를 구성한다.
- `pageTemplate=classic`은 타임라인과 기억 카드를 균형 있게 보여준다.
- `pageTemplate=album`은 기억 카드와 사진형 경험을 먼저 보여준다.
- `pageTemplate=letter`는 방명록과 남겨진 글을 먼저 보여준다.
- `pageTemplate=gallery`는 기억 카드를 여러 장 펼쳐 가족 앨범처럼 보여준다.
- `pageTemplate=story`는 기억 카드 한 장면을 크게 보여주고 넘겨보는 흐름을 강조한다.

## AI 태그 원칙

- AI는 고인의 삶을 대신 판단하지 않고, 유족이 편집하기 쉬운 태그 후보만 제안한다.
- 공개 페이지에 노출되는 태그는 유족이 직접 승인한 항목만 사용한다.
- MVP는 로컬 규칙 기반 추천으로 UX를 먼저 검증한다.
- 운영 버전에서는 생애 글, 공지사항, 기억 카드, 승인된 방명록을 LLM에 전달해 태그 후보와 짧은 요약 초안을 만든다.
- 반려동물 memorial도 같은 구조로 두되, 질병/사망 원인처럼 민감한 표현은 기본 추천에서 제외한다.

## 로그인 원칙

- MVP는 유족 코드와 memorial별 가족 권한으로 편집 권한을 보호한다.
- 운영 베타에서는 현재 데모 Google 버튼을 실제 OAuth 흐름으로 교체한다.
- 공개 페이지 조회와 방명록 작성은 로그인 없이 가능하게 유지한다.
- 로그인 사용자의 방명록 수정/삭제는 Google 사용자 ID를 우선 사용하고, 비로그인 작성자는 `ownerToken`으로 보호한다.
- 편집, 승인, 숨김, 고정, 편집 기록, QR 다운로드는 등록된 가족 권한 또는 유족 코드로 제한한다.
- Google 계정은 사람 식별 수단이고, 실제 권한은 memorial별 가족 권한 테이블에서 판단한다.

## 다음 개발 단계

- 데모 Google 로그인 버튼을 실제 OAuth/Spring Security로 교체
- 이미지 업로드 저장소
- QR 스캐너 실기기 판독 테스트
- 공개 페이지 slug
- 신고/차단/스팸 방지

## 후원/꽃 놓기 구상

후원 기능은 추모의 진정성을 해치지 않도록 "과시형 결제"가 아니라 "작은 마음을 남기는 행동"으로 설계한다.

초기 상품안:

- 작은 꽃: 4달러 또는 4,000원대
- 추모 꽃다발: 20달러 또는 20,000원대
- 유족에게 직접 송금되는 후원과 서비스 운영 후원은 분리 표시

원칙:

- 금액 순위를 노출하지 않는다.
- 후원자는 이름 공개, 익명 공개, 비공개를 선택한다.
- 유족이 후원 기능을 켠 memorial에서만 표시한다.
- 결제 전에는 반드시 "유족에게 전달되는 금액/수수료/서비스 운영비"를 명확히 표시한다.
- 결제 전에는 환불 가능 조건, 환불 불가 조건, 처리 예상 시간을 버튼 가까이에 표시한다.
- 환불된 꽃은 공개 목록에서 숨기되, 유족/관리자 기록에는 주문과 환불 이력을 남긴다.
- 고인의 페이지 본문보다 후원 UI가 더 강하게 보이지 않게 한다.

추천 MVP:

1. 실제 결제 전, "꽃 놓기" UI를 비활성 상태로 노출해 반응을 본다.
2. Stripe/토스페이먼츠 중 하나를 붙여 결제 intent를 만든다.
3. `memory_tribute_orders` 테이블에 꽃 종류, 금액, 메시지, 공개 여부, 결제/환불 상태를 저장한다.
4. 방명록과 별개로 "남겨진 꽃" 영역에 조용히 표시한다.

현재 프로토타입:

- 공개 생애 페이지에 "남겨진 꽃" 섹션을 노출한다.
- 작은 꽃, 추모 꽃다발 가격대를 보여주되 실제 결제는 발생하지 않는다.
- 방문자가 시범으로 꽃을 남기면 금액 순위 없이 조용한 카드로 표시한다.
- 다음 단계에서 결제 성공/환불 webhook과 `memory_tribute_orders` 저장으로 교체한다.
